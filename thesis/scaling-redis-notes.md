# โน้ตความเข้าใจ: ทำไมต้อง scale / Redis มาเกี่ยวตอนไหน

> ถอดจากโน้ตลายมือของ Pete (2026-07-18) — สรุปข้อถกเถียงเรื่อง rate limit, replica, Redis ของทีม

## 1. เหตุผลของการ scale FastAPI มี 2 ข้อ

**ข้อ 1 — รับ request เยอะ (throughput):** ถ้า request เยอะมาก ๆ pod เดียวจะทำไม่ไหว
*แต่ของเราอาจจะไม่ได้เยอะขนาดนั้น* — นักศึกษาทั้งคาบคือหลักร้อย request ซึ่ง pod เดียวรับสบาย

**ข้อ 2 — fault tolerance (ตัวจริง):** กันว่าถ้า pod ที่รัน FastAPI ตาย ระบบยังอยู่
request จะถูกส่งไป pod อื่นแทนโดยอัตโนมัติ (load balancer จัดการ ไม่ต้องทำเอง)

```
FastAPI A   ← Dead
FastAPI B   alive   ← ระบบยังทำงานต่อได้
FastAPI C   alive
```

นี่คือเหตุผลหลักของการ scale — ไม่ใช่เพราะกลัว request เยอะ แต่กลัว pod เดียวตายตอนคาบเรียน

## 2. ตัวอย่างห้องสอบ: ทำไมตัวนับ in-memory พังเมื่อมีหลาย pod

กติกา: นศ. 1 คน ขอออกห้องน้ำได้ **ไม่เกิน 2 ครั้ง** / อาจารย์คุมสอบ 3 คน (= 3 pods) ต่างคนต่างนับในหัว

| ครั้งที่ | อาจารย์ | ที่อาจารย์นับได้ | ผล |
|---|---|---|---|
| 1 | A | ปีเตอร์ 1 | อนุญาต |
| 2 | B | ปีเตอร์ 1 — *เพราะเขาไม่รู้ว่าขอ A ไปแล้ว!* | อนุญาต |
| 3 | C | ปีเตอร์ 1 | อนุญาต |
| 4 | A | ปีเตอร์ 2 แล้วนะ | อนุญาต |
| 5 | B | ปีเตอร์ 2 | อนุญาต |
| 6 | C | ปีเตอร์ 2 | อนุญาต |

เท่ากับตอนนี้เขาออกไปแล้ว **6 รอบ** ทั้งที่กติกาบอก 2 — limit บวมเป็น (2 × จำนวนอาจารย์)

## 3. ถ้าเป็น Redis: เขียนบนกระดานเดียวกัน

ทุกการขอต้องไปขีดนับบนกระดานกลางก่อน ไม่ว่าจะขอผ่านอาจารย์คนไหน

| ครั้งที่ | อาจารย์ | กระดานกลาง | ผล |
|---|---|---|---|
| 1 | A | ปีเตอร์ I | ผ่าน |
| 2 | B | ปีเตอร์ II | ผ่าน |
| 3 | C | ปีเตอร์ III — เกิน 2 | **โดนกัน** |

limit 2 เป็น 2 จริง เพราะผู้ตัดสินทุกคนใช้ตัวเลขก้อนเดียวกัน

## 4. ย้อนกลับมาที่ Virtual Lab: กด Launch ซ้ำ

```
launch รอบแรก  ──► [FastAPI A] ──► สร้าง session ให้
launch ซ้ำ     ──► [FastAPI B] ──► ✗ reject (มี session ที่ยังไม่จบอยู่แล้ว)
```

พฤติกรรมแบบนี้เรียกว่า **idempotency** — กดกี่ครั้งผลลัพธ์ก็เท่ากับกดครั้งเดียว

## 5. คำถามเปิดจากโน้ต: แล้วจะใช้วิธีไหนตัดสิน "launch ซ้ำ"?

ตัวเลือกที่จดไว้: (ก) FastAPI ใช้ status ใน database เป็นผู้ตัดสิน (ข) ใช้ Redis

**คำตอบที่ทีมสรุป: ใช้ database (ข้อ ก)** ด้วย partial unique index:

```sql
CREATE UNIQUE INDEX idx_sessions_one_live_per_lab
  ON sessions (user_id, lab_id)
  WHERE status IN ('pending','provisioning','running');
```

เหตุผลสำคัญ — **Postgres ก็เป็น "กระดานกลาง" เหมือนกัน**: ทุก pod เขียนลง database ก้อนเดียวกันอยู่แล้ว ดังนั้น unique index ใช้ได้ถูกต้องแม้มี 3 pods โดยไม่ต้องพึ่ง Redis เลย และมันแรงกว่า rate limit ด้วย เพราะไม่ใช่ "ช้าลงหน่อย" แต่เป็น "session ที่มีชีวิตมีได้ตัวเดียวต่อแลป จบ"

สรุปการแบ่งหน้าที่:

| ปัญหา | ผู้ตัดสิน | ทำไม |
|---|---|---|
| กด Launch ซ้ำซ้อน (idempotency) | **Postgres** (unique index) | session คือความจริงถาวร อยู่ใน DB อยู่แล้ว — DB เป็นกระดานกลางในตัว |
| กดถี่เกิน (rate limit เช่น 10 ครั้ง/ชม.) | ตอนนี้: in-memory (1 pod) / เมื่อ scale: **Redis** | ตัวนับเป็นของชั่วคราว หายได้ — งานถนัด Redis เมื่อมีหลาย pod |
| ใช้ชั่วโมงเกิน | **Postgres** (quota + wall-clock จาก sessions) | เกราะหลักตัวจริงของระบบ |
